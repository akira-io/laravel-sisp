# Idempotency

Idempotency prevents one checkout intent from creating more than one local SISP transaction when the same payment request is submitted more than once.

Payment flows are not single-step operations. A customer can double-click a button, refresh a page, lose connectivity, return from a failed SISP page, or retry after the browser has already posted the payment form. Without a stable application key, each POST to `sisp.payment` can look like a new checkout and the package cannot know whether the user meant "same checkout" or "new purchase".

## Core Terms

| Value | Owner | Stability | Purpose |
| --- | --- | --- | --- |
| `idempotency_key` or `checkout_intent_id` | Application | Stable for one checkout intent | Prevent duplicate local transactions |
| `merchant_ref` | Package generator | Unique at transaction level | Merchant payment reference sent to SISP |
| `merchant_session` | Package generator | Unique at attempt level | SISP session value used in the request fingerprint |

Do not use `merchant_ref` or `merchant_session` as the application idempotency key. They are SISP-facing values and remain generated by the configured generators. Use a separate application key that represents the checkout intent, such as a cart UUID, order UUID, checkout session UUID, or payment intent UUID created by your application before redirecting to payment.

## What Causes Duplicate Transactions

### Missing Checkout Intent Key

If the frontend posts to `sisp.payment` without `idempotency_key` or `checkout_intent_id`, the package must treat the request as a new payment request. A retry from the browser can then create a new transaction.

### New Key on Retry

If the application generates a new idempotency key each time the form is rendered or the user clicks "Pay", the package sees each submission as a new checkout. The key must be created once for the checkout and reused until that checkout reaches a terminal application decision.

### Failed Payment Followed by User Retry

A failed SISP attempt should not create a second local transaction. It should create a new attempt under the original transaction, keep the original transaction audit trail, and send SISP a fresh `merchant_session`.

### Generator Collisions

The package lets applications customize `merchant_ref`, `merchant_session`, and timestamp generators. A custom generator can return the same value twice. The package treats generated values as candidates and persists them only after database uniqueness constraints accept them.

### Non-Atomic Callback Writes

SISP can retry callbacks. The package records callback data on the attempt and, when the attempt should affect the parent transaction, writes through to the transaction. Those writes must be atomic. If the attempt were updated but the transaction update failed, future SISP retries could be blocked as already processed while the transaction stayed stale. The callback write paths run inside database transactions.

## Data Model

Idempotency is implemented through two supporting tables.

### `sisp_payment_intents`

This table reserves the application checkout key.

| Column | Meaning |
| --- | --- |
| `idempotency_key` | Unique application checkout key |
| `transaction_id` | Transaction created for this key, nullable while processing |
| `status` | `processing`, `submitted`, or `failed` |
| `failure_reason` | Captured exception message when transaction creation fails |

The unique index on `idempotency_key` is the primary duplicate guard. The reservation is made before creating a transaction.

### `sisp_transaction_attempts`

This table records each SISP submission attempt for a transaction.

| Column | Meaning |
| --- | --- |
| `transaction_id` | Parent transaction |
| `attempt_number` | Sequential attempt number for the transaction |
| `merchant_ref` | SISP merchant reference for this attempt |
| `merchant_session` | SISP merchant session for this attempt |
| `gateway_transaction_id` | Transaction ID returned by SISP callback |
| `payload` | Payment request sent to SISP |
| `callback_payload` | Callback payload received from SISP |
| `failure_reason` | Package failure reason for invalid callbacks |
| `superseded_at` | Marks an older attempt after a retry creates a newer attempt |

Attempts make retries auditable and let callbacks from older SISP attempts be processed safely.

## Payment Flow in v1

In v1, idempotency is handled by `CreateIdempotentPaymentTransactionAction`, which is called by `PaymentController`.

1. `PaymentController` validates the payment request.
2. `CreateIdempotentPaymentTransactionAction` reads the first non-empty request key from `config('sisp.idempotency.request_keys')`.
3. If no key is present, the package creates a normal unique payment transaction.
4. If a key is present, the package reserves it in `sisp_payment_intents`.
5. If reservation succeeds, the package creates the SISP payment transaction, stores the first transaction attempt, and marks the intent as `submitted`.
6. If reservation fails because another request already reserved the key, the package either reuses the existing transaction or returns a conflict while the first request is still processing.

The default request keys are:

```php
'idempotency_key'
'checkout_intent_id'
```

They are accepted by `StorePaymentRequest` as nullable strings up to 255 characters.

## Reservation States

### `processing`

The key has been reserved, but the package has not linked a transaction yet.

A second request with the same key receives:

- JSON request: HTTP `409` with `Payment is already being processed.`
- Browser form request: redirect back with a validation error

### `submitted`

The key is linked to a transaction.

When the same key is posted again:

- If the transaction is still payable and the package can reconstruct the original SISP request, the same payment form is returned.
- If the transaction failed and retry is allowed, the package creates a new attempt on the same transaction.
- If the stored payload is missing, the package returns a conflict instead of throwing an unexpected hydration error.

### `failed`

The package failed before linking a transaction. Failed reservations with `transaction_id = null` are reclaimable. A later request with the same key moves the row back to `processing` and tries again, which prevents transient server errors from permanently blocking the checkout key.

## Retry Behavior

A retry after a failed payment must not create a new transaction. The package does this instead:

1. Locks the transaction row with `lockForUpdate()`.
2. Generates a fresh SISP request from the existing transaction.
3. Creates a new `sisp_transaction_attempts` row.
4. Marks the previous current attempt with `superseded_at`.
5. Clears callback response fields on the transaction.
6. Sets the transaction back to `pending`.

This gives SISP a fresh `merchant_session` while preserving one local transaction and a complete attempt history.

## Callback Behavior

Callbacks are matched by `merchant_ref` and `merchant_session` against `sisp_transaction_attempts`.

- Current attempt: callback writes through to the transaction.
- Superseded failed attempt: attempt is recorded, transaction is not overwritten.
- Superseded completed attempt: callback is allowed to promote the transaction to completed.

This prevents a late failed callback from an old attempt from overwriting a newer retry that is still pending.

Callback attempt updates and transaction updates are wrapped in a single database transaction. If the transaction write fails, the attempt update is rolled back, so SISP can retry the callback without being blocked by a half-written attempt.

## Identifier Collision Handling

`merchant_ref` and `merchant_session` are generated by configurable classes:

```php
'generators' => [
    'merchantSession' => Akira\Sisp\Actions\Generators\MerchantSessionGeneratorAction::class,
    'merchantReference' => Akira\Sisp\Actions\Generators\MerchantReferenceGeneratorAction::class,
    'timeStamp' => Akira\Sisp\Actions\Generators\TimeStampGeneratorAction::class,
],
```

Because these generators are configurable, the package assumes they can collide. The database enforces uniqueness for:

- transaction `merchant_ref`
- attempt `merchant_session`
- attempt pair `merchant_ref` and `merchant_session`
- attempt pair `transaction_id` and `attempt_number`

When a collision happens during creation or retry, the package catches the unique-constraint failure and generates a new candidate up to the value exposed by `LoadConfig::getIdentifierGenerationMaxAttempts()`.

Between attempts, it sleeps for the value exposed by `LoadConfig::getIdentifierGenerationCollisionRetrySleepMicroseconds()`.

If all candidates collide, the package throws `UnableToGenerateUniquePaymentIdentifiersException`.

## Configuration

The default config enables idempotency:

```php
'idempotency' => [
    'enabled' => env('SISP_IDEMPOTENCY_ENABLED', true),
    'request_keys' => ['idempotency_key', 'checkout_intent_id'],
],
```

To disable package-level payment idempotency:

```env
SISP_IDEMPOTENCY_ENABLED=false
```

You can change accepted request key names in `config/sisp.php`:

```php
'idempotency' => [
    'enabled' => env('SISP_IDEMPOTENCY_ENABLED', true),
    'request_keys' => ['order_uuid', 'checkout_intent_id'],
],
```

Keep at least one stable key in your payment POST body.

## Practical Integration

Create the checkout key when your application creates the cart, order, or checkout session.

```php
$checkoutIntentId = (string) Str::uuid();

session()->put('checkout_intent_id', $checkoutIntentId);
```

Do not create a new key during each render. Do not create a new key in JavaScript on each click.

Send the key with form requests:

```blade
<form action="{{ route('sisp.payment') }}" method="POST">
    @csrf

    <input type="hidden" name="checkout_intent_id" value="{{ $checkoutIntentId }}">

    <input type="number" name="amount" required>
    <button type="submit">Pay</button>
</form>
```

Send the key with JSON requests:

```php
Http::post(route('sisp.payment'), [
    'checkout_intent_id' => $checkoutIntentId,
    'amount' => 6000,
    'items' => [
        [
            'product_name' => 'Ticket',
            'quantity' => 1,
            'unit_price' => 6000,
            'total_price' => 6000,
        ],
    ],
    'customer_email' => 'customer@example.test',
]);
```

Handle HTTP `409` as "same checkout is already in progress", not as a gateway failure. In an API flow, show a payment processing state and poll your own order endpoint. In a browser flow, use the validation error flashed by the package.

Use one idempotency key for one checkout intent. Do not reuse the same key for another cart, another order, a different amount, or a new customer payment after the old checkout has been abandoned. If the user intentionally starts a new checkout, create a new key.

## Operational Checklist

1. Run the package migrations so `sisp_payment_intents` and `sisp_transaction_attempts` exist.
2. Send `checkout_intent_id` or `idempotency_key` with each payment POST.
3. Keep the key stable across double-clicks, browser retries, and user retries for the same checkout.
4. Keep SISP generator customization deterministic enough to produce candidates, but let the package enforce uniqueness.
5. Treat HTTP `409` as "same checkout is already in progress", not as a gateway failure.
6. Use transaction attempts for audit and support.
7. Use status reconciliation for old pending transactions that never received a final SISP callback.

## Common Mistakes

### Using a Timestamp as the Checkout Key

Timestamps can collide under concurrent requests and usually change on retry. Use a UUID generated by your application instead.

### Reusing One Key for an Order After Changing the Amount

The key identifies the checkout payload. If the amount or item list changes, start a new checkout intent.

### Relying Only on Disabled Buttons

Frontend guards reduce double-clicks, but they do not cover network retries, browser resubmissions, or concurrent requests. Server-side idempotency is still required.

### Creating a New Local Transaction for SISP Retry

A failed retry should create a new attempt under the same transaction. Creating a new transaction breaks reconciliation and makes support workflows ambiguous.

## Recovery Notes

If a checkout is stuck:

1. Look up the row in `sisp_payment_intents` by `idempotency_key`.
2. If `status = processing` and no request is active, inspect application errors around `updated_at`.
3. If `status = failed` and `transaction_id` is null, the next request with the same key can reclaim it.
4. If `status = submitted`, inspect the linked transaction and its attempts.
5. If the transaction is pending and SISP has a final status, run status reconciliation.

Do not delete rows as the first response. The rows are part of the payment audit trail.
